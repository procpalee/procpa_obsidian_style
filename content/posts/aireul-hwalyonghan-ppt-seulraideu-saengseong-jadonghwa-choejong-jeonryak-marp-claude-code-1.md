AI를 활용하여 업무 효율성을 극대화할 수 있는 대표적인 분야 중 하나가 바로 PPT와 같은 슬라이드 제작입니다. 하지만 시중의 여러 AI 도구(Gamma, Canva, Claude design 등)를 사용해 보면 처음에는 놀라운 결과물에 감탄하지만, 시간이 지날수록 디자인의 일관성이 떨어지고, 결과물의 품질이 매번 달라져서 실무에 적용할 만한 수준의 결과물을 얻기는 쉽지 않습니다.

최근 이러한 한계를 극복해 보고자 **Claude Code**를 활용해 다양한 시도를 해보았고, 그 과정에서 **Marp**라는 도구를 활용한 워크플로우가 훌륭한 대안이 될 수 있음을 확인하여 소개해 드리고자 합니다.

## 1. AI 슬라이드 제작 방식 선택
요즘 AI 기술을 활용해 슬라이드를 제작해 주는 도구들이 쏟아져 나오고 있습니다. 시중의 거의 모든 도구를 직접 사용해 보며 겪은 시행착오를 바탕으로, 각 방식의 장단점과 이를 극복하기 위해 제가 선택한 전략을 소개해 드리고자 합니다.

이 도구들은 크게 **출력 포맷**과 **작동 방식**에 따라 두 가지 축으로 나누어 분석해 볼 수 있습니다.

### 1.1. 출력 포맷 : HTML이냐, PPT냐
가장 먼저 결정해야 할 요소는 바로 **출력 포맷**입니다.

**PPT(.pptx)** 파일은 회사 내부 보고나 클라이언트 전달용 등 반드시 파일 형태로 주고받아야 하는 특수한 상황에서 여전히 필요합니다. 하지만 그러한 제약이 없는 상황이라면, **HTML 슬라이드가 거의 모든 면에서 유리**합니다.

이유는 간단합니다. HTML과 CSS를 활용하면 현대적인 웹 디자인의 모든 요소를 슬라이드에 그대로 적용할 수 있기 때문입니다. 부드러운 그라데이션, 그리드 레이아웃, 코드 하이라이팅 등 PPT로 구현하려면 복잡하거나 불가능한 연출들을 CSS 몇 줄로 구현할 수 있습니다. 게다가 AI 모델들은 HTML/CSS 코드를 작성하는 능력이 매우 뛰어납니다. PPT의 복잡한 XML 구조보다 웹 표준 데이터가 훨씬 풍부하게 학습되어 있기 때문입니다.

### 1.2. 작동 방식 : 슬라이드 제작 방식의 세 가지 접근법
현재 시중의 슬라이드 제작 방식은 크게 **AI-Native 방식**, **프로그래밍 방식**, 그리고 이 둘의 장점을 결합한 **하이브리드(AI+프로그래밍) 방식**의 세 가지 범주로 분류할 수 있습니다.

| 분류 | 특징 | 제어권 | 진입 장벽 | 대표 도구 |
| --- | --- | --- | --- | --- |
| **AI-Native 방식** | 빠른 속도, 원클릭 생성 | 낮음 | 낮음 | Gamma, Canva, 젠스파크(Genspark) 등 |
| **프로그래밍 방식** | 고도의 디자인 일관성, 결정론적 렌더링 | 높음 | 높음 | Marp, Slidev, Reveal.js 등 |
| **하이브리드 방식** | AI의 코드 자동 생성 + CSS 규칙 통제 | 높음 | 낮음 | Claude Code + Marp (본 시스템) 등 |

#### 1.2.1. AI-Native 방식(AI 전용 툴)
AI 전용 툴은 사용자가 입력한 자연어 프롬프트를 바탕으로 AI가 문서 개요부터 레이아웃 설계, 이미지 삽입, 테마 디자인까지의 전 과정을 한 번에 해결해 주는 솔루션입니다.

- **장점**: 디자인 경험이 없는 실무자도 1~2분 안에 그럴듯한 발표 장표를 즉시 만들어낼 수 있습니다.
- **단점**: 결과물이 AI 모델의 생성 확률에 의존하므로 디자인 **일관성이 부족**하고 **결과물이 매번 달라집니다**. 특히 긴 원문 보고서의 흐름을 임의로 건너뛰거나 내용을 요약하는 한계가 있습니다.

#### 1.2.2. 프로그래밍 방식(코드 기반 도구)
마크다운(Markdown)과 같은 텍스트 문서 파일에 디자인 규칙을 담은 스타일시트(CSS 등)를 결합하여 슬라이드를 렌더링하는 방식입니다.

- **장점**: 내용(콘텐츠)과 형식(디자인)이 완전히 분리되어 있어 원문 텍스트의 맥락과 구조가 그대로 유지됩니다. 스타일이 **규칙**에 의해 정의되므로 항상 **일관성 있는 디자인**을 보장합니다.
- **단점**: 코드를 다뤄야 하므로 비개발자에게는 마크다운 문법과 CSS를 다루기 위한 **진입 장벽**이 존재합니다.

#### 1.2.3. 하이브리드 방식(AI + 프로그래밍 결합)
AI의 우수한 생산성과 프로그래밍 방식의 통제력을 결합한 최신 접근법입니다. 일반적으로 **Claude Code**나 **Codex** 등 코딩 에이전트에게 발표용 슬라이드 코드를 작성해 달라고 요청하는 형태가 이에 속합니다.

사용자가 프레젠테이션용 구조를 직접 작성하는 대신, AI가 이 코드를 대신 설계하고 빌드합니다. AI가 정해진 포맷(마크다운, HTML, 코드 템플릿 등)에 따라 코드를 구조화하여 작성해 주면, 최종 변환과 렌더링은 해당 툴의 엔진이 CSS 등의 디자인 규칙에 맞게 정적이고 결정론적으로 수행합니다. 결과적으로 빠른 제작 속도와 정밀한 제어력이라는 두 가지 장점을 동시에 가질 수 있습니다.

### 1.3. 최종 선택
Claude Code나 Codex 등을 활용하여 슬라이드 소스 코드를 생성하는 **하이브리드 방식**은 현재 가장 강력한 접근법입니다. 하지만 단순히 원본 원고를 던져주며 슬라이드로 변환해 달라고 요청하는 방식은 본질적으로 AI-Native 방식의 불확정성 한계를 고스란히 답습하게 됩니다.

이러한 문제를 해결하고 완벽한 디자인 일관성을 확보하기 위해, 저는 슬라이드 코드를 구성하는 **레이아웃 요소를 템플릿(컴포넌트) 형태로 세분화하여 사전에 규격화(CSS 클래스 등으로 정의)** 해 두는 것이 필수적이라고 판단했습니다.

이를 구체적으로 실현하기 위해 마크다운(Markdown) 기반으로 슬라이드를 렌더링하는 **Marp**를 핵심 도구로 채택했습니다. 그리고 원본 글을 분석하여 규격화된 Marp용 마크다운으로 변환해 주는 전용 **스킬**을 Claude Code 환경에 세팅함으로써, 클로드코드의 **커맨드** 하나만으로 작동하는 자동화 워크플로우 전략을 구축하게 되었습니다.

## 2. 마크다운 슬라이드 도구 비교와 Marp 선택 이유
대표적인 프로그래밍 기반 프레젠테이션 도구인 Marp, Slidev, Reveal.js, Quarto의 특징을 분석하고 장단점을 비교해 보았습니다.

| 도구 | 주요 기반 | 장점 | 단점/비개발자 장벽 |
| --- | --- | --- | --- |
| **Marp** | 순수 마크다운 | 단순하고 가벼움, 옵시디언 등 일반 마크다운 편집기와 완벽 호환 | 고급 인터랙션 및 애니메이션 제한 |
| **Slidev** | Vue.js + Vite | 개발자 친화적, 동적 웹 컴포넌트 및 코드 실시간 작동 지원 | Vue.js/npm 설치 등 프론트엔드 지식 필수, 비개발자에게 높은 진입 장벽 |
| **Reveal.js** | HTML + JavaScript | 웹 표준 기반의 자유로운 커스터마이징, 화려한 3D 트랜지션 | 기본이 HTML 마크업 중심이라 마크다운 작성 시 별도 설정 필요 및 높은 편집 공수 |
| **Quarto** | Pandoc + R/Python | 데이터 과학 최적화, 통계 패키지/보고서 연동 강력 | R, Python, Jupyter 등 복잡한 데이터 분석 환경 세팅 필요, 무거운 시스템 구조 |

여러 도구가 있었지만 Marp를 선택한 이유는 다음과 같습니다.

### 2.1. 옵시디언(Obsidian)과의 시너지
저는 모든 개인 지식 관리와 원고 작성을 옵시디언에서 마크다운 문서로 수행하고 있습니다. Marp는 복잡한 빌드 환경이나 문법 없이, 마크다운 파일 중간에 `---` 구분선만 추가하고 CSS 스타일을 매칭해 주면 발표용 HTML 슬라이드를 즉시 출력해 줍니다. 옵시디언에서 쓴 글을 조금만 가공하면 바로 발표 자료로 트랜스폼할 수 있다는 점에서 시너지가 높습니다.

### 2.2. 비개발자를 위한 접근성
저는 개발자가 아닌 현직 회계사입니다. 아무리 좋은 도구라도 사용법이 복잡하면 실무에서 지속적으로 활용하기 어렵습니다. Slidev, Reveal.js, Quarto 등은 훌륭한 기능을 제공하지만, npm 패키지 빌드나 Vue.js 컴포넌트 개발, 혹은 R/Python 가상환경 세팅 등 비개발자가 접근하기에 진입 장벽이 높은 기술적 배경을 요구합니다. 반면 Marp는 복잡한 개발 환경 구축 없이 단 한 줄의 CLI 명령어나 텍스트 에디터의 확장 프로그램(VS Code Extension 등)만으로 즉시 동작합니다. 덕분에 학습 비용을 최소화하고 바로 활용할 수 있었습니다.

## 3. Marp 슬라이드 자동화 워크플로우 구축기
원고 작성부터 최종 슬라이드 발행까지의 과정을 일원화하기 위해 다음과 같은 자동화 파이프라인(Pipeline)을 설계했습니다.

### 3.1. 파이프라인 기본 동작 원리
기본 동작 프로세스는 다음과 같이 세 단계로 순차 진행됩니다.

```
원본 마크다운 원고
    ↓  [Claude(Skills) — 구조 분석 및 레이아웃 클래스 매칭]
Marp용 마크다운 (.md 파일 및 Frontmatter 생성)
    ↓  [Marp CLI — 렌더링 및 빌드]
HTML 발표용 슬라이드 완성
```

[스크린샷: Claude Code 터미널에서 슬라이드 빌드 커맨드를 실행하여 HTML이 생성되는 과정]

이 과정은 단순해 보이지만, 이를 **일관되고 디자인적으로 전문적인 결과물**로 완성하려면 몇 가지 세부적인 구축 작업이 필요했습니다.

### 3.2. 워크플로우 구축 단계

#### 1단계: Design.md 벤치마크
가장 먼저 진행한 작업은 검증된 브랜드를 대상으로 **디자인 시스템을 벤치마킹**하는 것이었습니다.

이를 위해 [getdesign.md](https://getdesign.md)에서 글로벌 테크 기업(Notion, Vercel, Stripe, Apple 등)의 완성도 높은 디자인 가이드를 다운 받았습니다. 참고로 **`design.md`** 란 구글의 스티치(Stitch)에서 제안한 개념으로, 색상 팔레트, 타이포그래피 스케일, 간격(Margin/Padding) 등 핵심적인 **디자인 규칙(Design Tokens)을 표준화하여 정의한 명세서**입니다.

![image.png](https://cdn.jsdelivr.net/gh/procpalee/my_obsidian_images/img/20260525200925534.png)


#### 2단계: CSS 파일 및 레이아웃 세팅
**CSS 파일**은 마크다운 원고를 HTML 발표용 슬라이드로 변환하는 시각적 렌더링 룰입니다. 웹 사이트 설계와 마찬가지로 색상, 글꼴 크기, 컴포넌트 여백 등을 **클래스(Class)** 로 조밀하게 정의합니다. 벤치마킹한 `design.md` 명세서를 기반으로 뼈대가 되는 기본 스타일시트와 샘플 HTML 결과물을 제작하도록 **Claude Code**에 요청했습니다. 최종 도출된 시안 중 저의 주 사용 목적인 발표에 가장 잘 어울린다고 판단한 **Notion 스타일**을 기본 테마로 선정했습니다.

베이스 테마 확정 이후 가장 핵심적이었던 과제는 **공통 컴포넌트와 전용 레이아웃의 완성**이었습니다. 이 프로세스에서도 Claude Code와 작성된 코드를 상시 리뷰하며 필요한 레이아웃 구조와 벤치마킹할 웹 디자인 레퍼런스를 실시간으로 제안해 퀄리티를 끌어올렸습니다. 

이렇게 레이아웃을 최대한 많이 설계하는 데는 분명한 목적이 있습니다. 슬라이드를 새로 생성할 때마다 AI가 기분에 따라 디자인을 즉흥적으로 창조하도록 방치하는 것이 아니라, 검증이 끝난 **정형화된 컴포넌트 풀(Pool)** 에서 본문 구조에 매칭되는 클래스만 선택해 정적으로 렌더링하기 위함입니다. 이 통제 방식을 통해 흔들림 없는 **디자인 일관성**을 확보하고 변환 소요 시간 또한 절감할 수 있습니다.

#### 3단계: 마크다운 변환 스킬 제작
디자인 시스템이 준비되었어도 원문을 일일이 Marp 규격에 맞춰 손으로 수정한다면 생산성이 떨어집니다. 이 과정을 자동화하기 위해 Claude 전용 **`md-to-marp` 스킬**을 제작했습니다.

이 스킬은 원본 텍스트를 분석하여 다음 작업을 처리합니다.

- 글의 맥락과 위계(H1~H3, 본문, 리스트)를 파악합니다.
- 각 슬라이드의 분량에 맞게 `---` 구분선으로 슬라이드를 나눕니다.
- 26종의 레이아웃 중 콘텐츠 특성에 가장 잘 맞는 CSS 클래스를 매칭합니다. (예: 3개 요소 비교는 `grid-3`, 통계 수치는 `stats` 클래스 부여)
- 문서 상단에 Marp 실행에 필요한 프론트매터(Frontmatter)를 주입합니다.

#### 4단계: Claude Code 기반 원클릭 파이프라인 완성
각기 분리되어 있던 변환, 렌더링, 검증 단계를 하나로 묶어 명령어 한 줄로 동작하게 만들었습니다. Claude Code의 커스텀 커맨드 기능을 이용해 단순화했습니다.

`/marp <파일명.md> [발표 테마] 이 명령어를 입력하면 다음 과정이 실행됩니다.

- **[1] 구조 변환**: `md-to-marp` 스킬을 호출해 Marp 마크다운으로 가공합니다.
- **[2] HTML 빌드**: Marp CLI를 백그라운드에서 호출하여 HTML 파일을 빌드합니다.
- **[3] 디자인 검증**: 독립된 컨텍스트로 구동되는 검증 에이전트(`marp-deck-reviewer`)가 빌드 결과물의 레이아웃 깨짐이나 텍스트 오버플로우를 감수(QA)합니다.
- **[4] 예외 처리**: 디자인 결함이 발견될 경우, 최대 2회까지 레이아웃 클래스를 보정하여 재빌드를 수행합니다.

## 4. 최종 결과물 및 향후 계획

### 4.1. 최종 결과물
앞으로 개인적인 발표에 활용할 슬라이드는 이번에 구축한 Notion 스타일 테마를 기반으로 제작할 계획입니다. 디자인은 주관적인 영역이지만, 텍스트의 전달력을 극대화해 주는 Notion 특유의 깔끔하고 구조적인 형태가 개인적으로 만족스럽습니다. 샘플 HTML 파일을 공유해 두었으니, 향후 Marp 사용을 고려하시는 분들께 좋은 참고가 되기를 바랍니다.

아울러 CSS 소스 코드는 GitHub에 공개해 두었습니다. 앞으로도 새로운 디자인 레이아웃과 컴포넌트를 추가하며 지속적으로 고도화해 나갈 예정입니다.

https://github.com/procpalee/Marp_Templates

### 4.2. 향후 계획
현재 안정화된 슬라이드 자동화 시스템을 디딤돌 삼아 두 가지 방향으로 고도화를 준비하고 있습니다.

#### 4.2.1. SNS 카드뉴스용 테마 추가
슬라이드에 적용된 레이아웃 시스템을 인스타그램, 스레드(Threads), 링크드인(LinkedIn) 등 소셜 미디어 플랫폼에 적합한 카드뉴스 포맷으로 확장할 예정입니다. 16:9 규격 외에 1:1 카드뉴스 규격을 생성하는 모드를 얹고, 빌드 완료된 PNG 이미지를 각 SNS API를 통해 원클릭으로 게시하는 파이프라인을 연계할 계획입니다.

이를 통해 글 한 편을 작성하는 것만으로 발표 슬라이드, 세로형 카드뉴스, SNS 캡션까지 유기적으로 파생 생산되는 원소스 멀티유즈(OSMU) 워크플로우를 만들고자 합니다.

#### 4.2.2. 실무형 제안서 및 보고서 컴포넌트 추가
현재의 슬라이드는 발표용 장표 중심이지만, 향후에는 제안서나 평가보고서 등에 활용할 수 있는 다양한 레이아웃을 추가할 예정입니다. 
텍스트 위주의 슬라이드를 넘어 정량적 데이터 보고서까지 이 파이프라인을 태워 제작할 수 있도록 CSS 컴포넌트를 확장해볼 계획입니다.
